# Language bindings

One Rust core (`q-periapt-ffi`, the C ABI), runtime-tested faces **C**, **WASM**,
**Swift**, and **Kotlin/JVM**, plus a package-tested **Android AAR/JNI** surface.
The headline property is *byte-identical results across platforms* — so runtime
bindings are validated against one shared oracle.

## Cross-platform consistency oracle

[`shared-test-vectors.json`](shared-test-vectors.json) is generated from the Rust
core:

```sh
cargo run -p q-periapt-backends --example refvec > bindings/shared-test-vectors.json
```

It contains a full ContextBound vector (keys, randomness, ciphertexts, and the
resulting 32-byte `secret`). The Rust/C-ABI, Swift, WASM, and Kotlin test suites
**`decapsulate(...)`** the vector's inputs and assert the result equals `secret`;
Swift and Kotlin also **`encapsulate(...)`** with the vector randomness and assert
`ct_pq`, `ct_trad`, and `secret` byte-for-byte. If any platform disagrees by a
single byte, the suites fail — that is the cross-platform guarantee, made executable.
For `CompatXWing`, the C/WASM/Swift/Kotlin faces expose the separate
ML-KEM-768 X-Wing seed keypair path; expanded ML-KEM secret keys are intentionally
ContextBound-only.

The C smoke test intentionally stays a link-level ABI check: it exercises a
representative keypair -> encapsulate -> decapsulate roundtrip and status-code
conformance, while the wider vector set is covered by the Rust/Swift/WASM/Kotlin
faces.

## Artifacts

- C header: [`../crates/q-periapt-ffi/include/q_periapt.h`](../crates/q-periapt-ffi/include/q_periapt.h)
  (generated by cbindgen; regenerate with the command in its banner).
- Static/shared lib: `cargo build -p q-periapt-ffi --release` →
  `target/release/libq_periapt_ffi.{a,dylib,so}`.

## Status

| Face | Consistency vs shared vector | How |
|---|---|---|
| **Rust core** | ✅ source of truth | `cargo test` (combiner KATs, X-Wing KAT) |
| **C ABI** (`q-periapt-ffi`) | ✅ verified | `cargo test -p q-periapt-ffi` (`ffi_matches_shared_vector`) |
| **Swift** (`swift/`) | ✅ verified | `swift test` plus `artifact/swift-xcframework.sh` binaryTarget consumer (CI: `bindings-swift` on macOS) |
| **WASM** (`q-periapt-wasm`) | ✅ logic verified + builds wasm32 | `cargo test -p q-periapt-wasm`; CI builds `wasm32` |
| **Kotlin** (`kotlin/`) | ✅ verified | `gradle test` (Panama FFM, JDK ≥22); CI: `bindings-kotlin` |
| **Android** (`android/`) | 🟡 package verified | `artifact/android-aar.sh` builds a deterministic AAR with four JNI/native ABI slices and an isolated Java consumer compile; CI: `bindings-android-aar`; runtime instrumentation still required |

**All five faces decapsulate `shared-test-vectors.json` to the same 32-byte secret,
byte-for-byte** — verified locally (Rust/C/Swift/WASM-on-Node/Kotlin) and gated in
CI; Swift and Kotlin additionally reproduce vector-driven encapsulation
byte-for-byte. The same five faces also reproduce the **combiner reference vectors**
([`contextbound-vectors.txt`](contextbound-vectors.txt), generated by
`cargo run -p q-periapt-ffi --example gen_combine_vectors`): each face feeds the
serialized `input` blob to the raw `q_periapt_combine` and gets the pinned 32-byte
key — so both the full hybrid *and* the bare combiner are byte-identical everywhere.
Kotlin needs a JDK ≥22 (stable FFM):

```sh
cargo build -p q-periapt-ffi --release
JAVA_HOME=/path/to/jdk22+ gradle -p bindings/kotlin test
```

Android AAR/JNI packaging needs an Android SDK/NDK and Rust Android targets:

```sh
rustup target add aarch64-linux-android x86_64-linux-android armv7-linux-androideabi i686-linux-android
sh artifact/android-aar.sh
```

This proves package shape, Android ELF slices, JNI symbols, dex conversion, and an
isolated Java consumer compile. It is not Android runtime proof until an emulator
or physical device runs instrumentation against the AAR.
