# Language bindings

One Rust core (`q-periapt-ffi`, the C ABI), current runtime-tested faces **C**,
**WASM**, and **Swift**, a source-migrated **Kotlin/JVM** face awaiting JDK 22,
plus a package-tested **Android AAR/JNI** surface.
The native ABI 2 product property is *one policy-controlled implementation across
platforms*: C, Swift, and Kotlin consume the same exact-nine dynamic
`q_periapt_*` contract; Android's JNI adapter consumes that native contract without
reimplementing cryptography. Static archives constrain the same reserved public
namespace but retain unsupported hidden bridge link symbols, so static embedding is
a trusted same-address-space boundary rather than access control. All public faces
reject raw/deterministic product bypasses. Byte-exact KATs remain in the Rust test layer.

These bindings expose the implemented KEM/policy surface only. They do not expose
identity directories, prekeys, ratchet state, message encryption, multi-device
management, or recovery. The future Continuity plan requires new state/wire vectors
and must not be inferred from the current shared-secret oracle; see
[`../docs/CONTINUITY_RESEARCH.md`](../docs/CONTINUITY_RESEARCH.md).

## Product and conformance evidence

[`shared-test-vectors.json`](shared-test-vectors.json) is generated from the Rust
core:

```sh
cargo run -p q-periapt-backends --example refvec > bindings/shared-test-vectors.json
```

It remains a Rust-internal deterministic ContextBound oracle, alongside the X-Wing
and combiner vectors. ABI 2 intentionally does not export the seeds, coins, raw
hybrid or combine calls needed to replay those vectors from an application.

The product smokes instead resolve the checked-in ML-DSA-65 signed policy, assert
its exact digest, reject ABI 1's version-only state, obtain key and encapsulation
randomness from the OS, round-trip under the policy-bound application context, and
exercise rollback/tamper/output-atomicity controls. This preserves meaningful
cross-language behavior without freezing a policy-bypass conformance API.

## Status

| Face | Consistency vs shared vector | How |
|---|---|---|
| **Rust core** | ✅ source of truth | `cargo test` (combiner KATs, X-Wing KAT) |
| **C ABI** (`q-periapt-ffi`) | ✅ host product smoke + contract verified | `bindings/c/smoke.c`; `artifact/c_abi_contract.py` |
| **Swift** (`swift/`) | ✅ host + XCFramework product package verified | `swift test`; physical device evidence is a separate source-bound gate |
| **WASM** (`q-periapt-wasm`) | ✅ logic verified + builds wasm32 | `cargo test -p q-periapt-wasm`; CI builds `wasm32` |
| **Kotlin** (`kotlin/`) | 🟡 source migrated; JDK 22 verification required | `gradle test` (Panama FFM, JDK ≥22) |
| **Android** (`android/`) | 🟡 ABI2 four-ABI AAR/package verified; ART runtime pending | `artifact/android-aar.sh`; `artifact/android-device-smoke.sh` |

Kotlin needs a JDK ≥22 (stable FFM):

```sh
cargo build -p q-periapt-ffi --release
JAVA_HOME=/path/to/jdk22+ gradle -p bindings/kotlin test
```

Android AAR/JNI packaging needs an Android SDK/NDK and Rust Android targets:

```sh
rustup target add aarch64-linux-android x86_64-linux-android armv7-linux-androideabi i686-linux-android
sh artifact/android-aar.sh
```

This proves package shape, Android ELF slices, JNI symbols, dex conversion, and an
isolated Java consumer compile. It is not Android runtime proof until an emulator
or physical device runs instrumentation against the AAR.
