# C-ABI link smoke test

A real C program that **links against the built `q-periapt-ffi` library** and calls the
exported functions — proving the C calling convention works end-to-end, not just that the
header parses. This complements the Swift/Kotlin/WASM faces (which exercise the same ABI
through their own FFI layers) with a plain-C consumer.

## What it checks ([`smoke.c`](smoke.c))

| | Check | Proves |
|---|---|---|
| 0 | `q_periapt_abi_version()`, `q_periapt_version()`, `q_periapt_fixed_suite_id()`, and `q_periapt_status_name()` match the compiled header/runtime contract | consumers can fail closed on ABI or suite drift before using cryptographic entry points |
| A | `q_periapt_combine()` reproduces a `ContextBound` golden vector byte-for-byte (from [`../contextbound-vectors.txt`](../contextbound-vectors.txt), same file the Swift/Kotlin/WASM faces pin) | correctness survives the C ABI. *That file is generated by `combine()` itself, so this is a cross-implementation **regression** oracle — it catches ABI/link/compiler corruption and cross-language drift, not an error in the single Rust core.* |
| B | `mlkem768_keypair` + `x25519_keypair` → `hybrid_encapsulate` → `hybrid_decapsulate`, asserting the two 32-byte secrets agree | the ContextBound expanded-key ABI surface (pointers, `size_t` lengths, `uint8_t`/`uint32_t` scalars, `int32_t` status) |
| B2 | `mlkem768_xwing_keypair` + `x25519_keypair` → `CompatXWing` encap/decap | the seed-dk X-Wing-compatible key format is usable through the C ABI, while expanded ML-KEM keys remain ContextBound-only |
| C | a `NULL` pointer returns `Q_PERIAPT_ERR_NULL`; a wrong output length returns `Q_PERIAPT_ERR_LENGTH` | the length-checked, non-aborting status-code contract holds across the boundary |

Exit code `0` = all passed.

## Run it

**Windows (MSVC)** — links `q_periapt_ffi.dll` via its import library; locates the toolchain
through `vswhere`, so `cl.exe` need not be on `PATH`:

```bat
bindings\c\build-and-run.bat
```

**macOS / Linux** (clang or gcc) — links the cdylib with an embedded rpath:

```sh
sh bindings/c/build-and-run.sh
```

Both scripts build `q-periapt-ffi` (`--release`), compile `smoke.c` against
[`../../crates/q-periapt-ffi/include/q_periapt.h`](../../crates/q-periapt-ffi/include/q_periapt.h),
link the resulting library, and run the test.

## Release Archive Smoke

For a download-style C consumer proof, run from the repo root:

```sh
sh artifact/c-package.sh
```

That gate builds a versioned host archive under `target/qperiapt-c-abi/`, containing
`include/q_periapt.h`, static/shared libraries, dynamic and static pkg-config files, CMake package
config, project license texts, CycloneDX CBOM/SBOM, `MANIFEST.json`, `SHA256SUMS`, and an archived
copy of this `smoke.c`. It then extracts the archive into a fresh prefix, verifies checksums,
safe archive layout, exact manifest file coverage, and BOM/license schema, then compiles/runs the smoke
through dynamic pkg-config, static pkg-config, `QPeriapt::qperiapt`, and
`QPeriapt::qperiapt_static`. All consumers compile with warnings-as-errors and must print
`ALL PASS`.

## Status

Verified **2026-06** on two OS/arch/compiler combos — **Windows 11 / x86_64-pc-windows-msvc /
MSVC** (links `q_periapt_ffi.dll`) and **macOS / aarch64 / clang** (links the dylib): the
smoke checks passed then, with byte-identical `combine()` output (Test A) on both. The
runtime metadata, CompatXWing seed-dk, and host C archive checks were added later and are covered by
the current test commands. CI is configured to run this test — the `windows` job runs
`build-and-run.bat`, the `check` job runs `build-and-run.sh` plus `artifact/c-package.sh` on Linux —
but this repo has **no git remote**, so CI has not executed; local runs are the current evidence.
(Broad cross-ISA assurance comes from the Rust `cross` job, not this C test.)
